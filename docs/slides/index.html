Problem

一个尴尬的场景

用户：
"帮我查一下这个月的市场分析报告。"

Agent 找到一篇付费文章，$0.002 一次。

但它没法买。

为什么不行？

没法注册账号
每个供应商都要 KYC，agent 跑不过
微支付不经济
$0.002 的金额，传统卡组织手续费比它大
实时性冲突
每次请求当场结清 ≠ 月结模式
供应方不愿建账户体系
为 agent 单独搭一套 = 巨大开发成本
Protocol

解法：复活一个 26 年前的 HTTP 状态码

HTTP 1999 年保留了状态码 402 Payment Required，几乎没人用过。
Coinbase 2024 提出 x402 协议，把支付塞进 HTTP 本身：三步握手。

# Step 1+2: client tries → server demands payment
GET /api/article
HTTP/1.1 402 Payment Required
{ accepts: [{ amount: "1000", payTo: "0xF72b...", asset: USDC }] }

# Step 3+4: client signs EIP-3009 → retries with X-PAYMENT → server settles → 200
GET /api/article
X-PAYMENT: <base64(signature)>
HTTP/1.1 200 OK + content
X-PAYMENT-RESPONSE: <tx hash>

无账号 · 无 API key · 每次请求当场结清

Building Block

关键拼图：EIP-3009

普通 ERC20 transfer 要求 payer 自己付 gas + 等链上确认。Agent 跑不动。
EIP-3009 引入 transferWithAuthorization：payer 只签名，不上链。

核心机制

离线签名
payer 用私钥对类型化数据签 EIP-712，得到 (v, r, s)
谁调谁付 gas
facilitator 拿签名上链 → USDC 自动扣 from → 转 to
nonce + 时间窗
每个签名只能用一次（防重放），过期作废

合约函数签名

function transferWithAuthorization(
  address from,        // payer
  address to,          // seller
  uint256 value,
  uint256 validAfter,
  uint256 validBefore,
  bytes32 nonce,
  uint8 v, bytes32 r, bytes32 s
) external

对 agent 意味着：付款像发 email 一样，签一张授权单，由 seller 兑付。

Challenge

新问题：Agent 哪来的私钥？

Agent 是服务端进程。让它直接持有私钥风险太大：

  • 泄漏一次 = 资产清零
  • Agent 有 prompt injection 攻击面
  • 容器镜像可能被反编译
  • Audit / 合规要求私钥隔离

需要的设计： 让 Agent 有支付能力，
但不持有私钥。

签名能力托管在受 IAM 鉴权
保护的后端服务里。

↓ 这正是 AgentCore Payments 的设计

AgentCore Payments

AWS 的托管签名 + 预算管控 + 审计方案

AgentCore Payments — 一句话总结

AWS 在 2026 年 5 月 推出的 Bedrock AgentCore 子服务：
为 AI Agent 托管支付凭证 + 签名钱包交易 + 审计资金流向，
原生支持 x402 协议。

托管
CDP 钱包凭证存在 AWS Secrets Manager，调用方零接触原始 secret
预算
PaymentSession.maxSpendAmount 硬限制，超额直接拒绝
审计
每笔 ProcessPayment 调用都有 IAM 身份记录 + 链上可追溯

支持 region：us-west-2（Preview 当前唯一可用区）

四层对象模型

每层都是上一层的具体化。配置时按从上到下顺序创建。

ProcessPayment 内部发生了什么

调用方写的代码


dp.process_payment(
  paymentManagerArn=...,
  paymentSessionId=...,    # 预算控制
  paymentInstrumentId=..., # 用哪个钱包签
  paymentInput={"cryptoX402": {
    "payload": x402_payload
  }}
)

注意：这一步不上链！只生成签名 proof，
上链是 seller 侧 Lambda@Edge 做的事。

服务端 6 步顺序执行

1
鉴权调用方有 ProcessPaymentRole 吗？
2
预算Session 还剩多少额度？
3
取 credentialRetrievalRole assume → Secrets Manager 取 CDP keys

构造 EIP-3009from / to / value / nonce / 时间窗
5
CDP 代签用 walletSecret 做 EIP-712 签名
6
返回 PROOF_GENERATED{ signature, authorization }

SDK / Python

boto3 没有 create_payment_*
→ 升 ≥ 1.43，需要 Python ≥ 3.10

CDK / 部署

Stack 跑到错的 region
→ 显式传 AWS_REGION=us-west-2

create_payment_manager AccessDenied
→ 跑 01_patch_iam_role.sh

API Schema

QUICKSTART 字段过时
→ EMBEDDED_CRYPTO_WALLET / linkedAccounts / expiryTimeInMinutes

CDP 配置

create_instrument 500
→ 启用 Delegated Signing + 真实 end user 邮箱

"No accounts found"
→ 用 Email OTP 而非 Google OAuth

Runtime / Bedrock

Architecture incompatible
→ --platform linux/arm64

Invocation 422
→ /invocations 改读 raw body

on-demand throughput 不支持
→ 用 us.anthropic.claude-...

端到端 / 钱

retry 时 403 Forbidden
→ retry 改用 GET（CloudFront 不允许 POST）

Faucet 钱"消失"
→ 钱要打到 payer 钱包，不是 seller

这套系统不解决的问题

✓ 协议解决了

重复扣款
EIP-3009 nonce 防重放
超额支出
Session 预算硬限制
密钥泄漏
Secrets Manager + Delegated Signing
审计追溯
链上 + IAM 双轨

✗ 协议不解决

退款 / 争议
链上不可逆，没有 RefundAPI
内容质量保证
付了不一定满意
定价发现
402 给的是 take-it-or-leave-it
合规 KYC/AML
主网各地法规不同

退款的解法：seller 应用层主动发反向 transfer，或上 escrow 合约。"链上即终局"是设计哲学。

这件事的意义

过去 20 年，互联网上所有收费都绑在账户上：注册、KYC、绑卡、订阅。

x402 + AgentCore Payments 在做不一样的事：

让支付变成一次 HTTP 请求级别的原语

内容方
只需一个链上地址，无需账户体系
Agent
带个支付 header 即可，无需预先注册
微支付
亚美分级经济上可行

这不是替换现有支付体系——是为机器对机器的经济打地基。
