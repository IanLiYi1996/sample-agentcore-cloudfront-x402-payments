一个尴尬的场景

你跟 AI 说：
"帮我查一下这个月的市场分析报告。"

Agent 去搜索，找到一篇付费文章，$0.002 一次。

但它没法买。

为什么不行？
  • 没法注册账号（每个供应商都要 KYC）
  • $0.002 的微支付，传统卡组织手续费比金额大
  • Agent 要在每次请求粒度付款，不是月结
  • 供应方不想为 agent 单独搭账户体系

这是 AI Agent 经济迈不过去的一道坎

解法：复活一个 26 年前的 HTTP 状态码

HTTP 1999 年就保留了一个几乎没人用过的状态码：

402 Payment Required

Coinbase 2024 提出 x402 协议，把支付塞进 HTTP 本身： 


Agent:    GET /api/article
Server:   HTTP/1.1 402 Payment Required
          { accepts: [{ amount: "1000", payTo: "0xF72b...", asset: USDC }] }

Agent:    (本地签一张 EIP-3009 授权单)

Agent:    GET /api/article
          X-PAYMENT: <base64 签名>
Server:   HTTP/1.1 200 OK + 内容
          X-PAYMENT-RESPONSE: <tx hash>

三步握手。无账号、无 API key、每次请求当场结清。

关键拼图：EIP-3009

如果用普通 ERC20 transfer：payer 要自己付 gas + 等链上确认。Agent 跑不动。

EIP-3009 的 transferWithAuthorization

核心机制：

  • payer 只签名，不上链
  • 任何人拿着签名 → 调合约 → USDC 自动转账
  • 谁调用谁付 gas（通常是 facilitator）
  • nonce + 时间窗防重放和过期

对 agent 意味着：

付款像发 email 一样——签一张授权单，由 seller 兑付。


function transferWithAuthorization(
  address from,        // payer
  address to,          // seller
  uint256 value,
  uint256 validAfter,
  uint256 validBefore,
  bytes32 nonce,
  uint8 v, bytes32 r, bytes32 s
) external

新问题：Agent 哪来的私钥？

Agent 是服务端进程。让它直接持有私钥风险太大：

  • 泄漏一次 = 资产清零
  • Agent 有 prompt injection 风险
  • 容器镜像可能被反编译
  • Audit / 合规要求私钥隔离
需要的是这种东西：
让 Agent 有支付能力，但不持有 key。 签名能力托管在受 IAM 鉴权保护的后端。

AgentCore Payments

AWS 的托管签名 + 预算 + 审计方案

AgentCore Payments — 一句话

AWS 在 2026 年 5 月 推出的 Bedrock AgentCore 子服务，
帮 AI Agent 托管 支付凭证 + 签名 钱包交易 + 审计 资金流向，
原生支持 x402 协议。

🔐 托管
CDP 钱包凭证存在 AWS Secrets Manager，调用方零接触
💰 预算
PaymentSession.maxSpendAmount 硬限制，超额直接拒绝
📜 审计
每笔 ProcessPayment 调用都有 IAM 身份记录 + 链上可追溯

四层对象模型

┌─────────────────────────────────────────────────────────┐ │ PaymentCredentialProvider │ │ ─ 把 CDP keys 存进 AWS Secrets Manager │ │ ─ 调用方拿不到原始 secret │ └────────────────────┬────────────────────────────────────┘ ▼ ┌─────────────────────────────────────────────────────────┐ │ PaymentManager (authorizerType=AWS_IAM) │ │ ─ 关联 ResourceRetrievalRole │ │ ─ IAM 鉴权决定谁能调 ProcessPayment │ └─────────────┬─────────────────────┬─────────────────────┘ ▼ ▼ ┌─────────────────────┐ ┌─────────────────────────────┐ │ PaymentConnector │ │ PaymentSession │ │ ─ 绑定 vendor=CDP │ │ ─ per-user 预算 │ │ ─ 关联 provider │ │ ─ maxSpendAmount=$1.0 │ └──────────┬──────────┘ │ ─ expiryTimeInMinutes=480 │ ▼ └─────────────────────────────┘ ┌─────────────────────────────────────────────────────────┐ │ PaymentInstrument (EMBEDDED_CRYPTO_WALLET) │ │ ─ 真实的钱包实体 │ │ ─ linkedAccounts=[email] → 绑到 CDP end user │ │ ─ 返回 walletAddress + WalletHub onboarding URL │ └─────────────────────────────────────────────────────────┘

ProcessPayment 内部发生了什么


dp.process_payment(
    paymentManagerArn=...,
    paymentSessionId=...,      # 预算控制
    paymentInstrumentId=...,   # 用哪个钱包签
    paymentInput={"cryptoX402": {"payload": x402_payload}}
)
服务端 (AgentCore Payments) 顺序执行： 1. 鉴权 ── 调用方有 ProcessPaymentRole 吗？ 2. 预算 ── PaymentSession 里还剩多少额度？ 3. 取 credential ── ResourceRetrievalRole assume → Secrets Manager 取 CDP keys 4. 构造 EIP-3009 authorization ── from / to / value / validAfter / validBefore / nonce 5. 让 CDP delegated signing 后端用 EIP-712 签名 (用 walletSecret) 6. 返回 { status: "PROOF_GENERATED", payload: { signature, authorization } } ⚠️ 这一步不上链！只生成签名 proof。

部署流程（7 步）

1 CDP 配置 ── 拿 API key + walletSecret，启 Delegated Signing，配 CORS

2 克隆代码 ── 设环境变量 CDP_API_KEY_ID 等

3 Seller CDK us-east-1 ── CloudFront + Lambda@Edge（Lambda@Edge 强制 us-east-1）

4 Payer CDK + 补 IAM us-west-2 ── 补 CDK 漏掉的 CreateWorkloadIdentity 等权限

5 建 Payments 资源 ── 5 子步：建 CDP end user → provider → manager → connector → instrument → session → WalletHub 授权 → faucet

6 部署 Agent ── docker buildx --platform linux/arm64

7 端到端测试 ── 看 payer/seller 链上余额变化

首次部署 1-2 小时（CloudFront 分发慢 + arm64 build emulation 慢）

这套系统不解决的问题

✅ 协议解决了

  • 重复扣款 ─ EIP-3009 nonce 防重放
  • 超额支出 ─ Session 预算硬限制
  • 密钥泄漏 ─ Secret 存 Secrets Manager + Delegated Signing
  • 审计追溯 ─ 链上 + IAM 双轨

❌ 协议不解决

  • 退款 / 争议 ─ 链上不可逆，没有 RefundAPI
  • 内容质量保证 ─ 付了不一定满意
  • 定价发现 ─ 402 给的是 take-it-or-leave-it
  • 合规 KYC/AML ─ 测试网随意，主网各地不同
退款的解法：seller 应用层主动发反向 ERC20 transfer，或上 escrow 合约让钱先入托管 + 争议期。 AgentCore Payments 的设计哲学是"链上即终局"——把退款明确推给应用层。

这件事的意义

过去 20 年，互联网上所有收费都绑在账户上：注册、KYC、绑卡、订阅。

x402 + AgentCore Payments 在做不一样的事：

让支付变成一次 HTTP 请求级别的原语

  • 内容提供者只要一个链上地址，无需账户体系
  • 消费者带个支付 header 即可，无需预先注册
  • Agent 之间互相调用可以实时结清，不需要平台撮合
  • 微支付（亚美分级）经济上可行

这不是替换现有支付体系，是为机器对机器的经济打地基。
