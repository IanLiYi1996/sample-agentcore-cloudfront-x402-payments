09 / 33 · PIECE 1 Protocol

x402 — 复活一个 26 年前的 HTTP 状态码

HTTP 1999 年保留了 402 Payment Required，几乎没人用过。Coinbase 2024 发布 x402，把支付塞进 HTTP 本身——一次交互内完成。

# Step 1+2: client tries → server demands payment
GET /api/article
HTTP/1.1 402 Payment Required
{ accepts: [{ amount: "1000", payTo: "0xF72b...", asset: USDC }] }
# Step 3+4: signs EIP-3009 → retries → server settles → 200
GET /api/article
X-PAYMENT: <base64(signature)>
HTTP/1.1 200 OK + content
X-PAYMENT-RESPONSE: <tx hash>

无账号 · 无 API key · 每次请求当场结清 · USDC on Base ≈ 200 ms · gas ≈ $0.0001

10 / 33 · BUILDING BLOCK EIP-3009

Payer 只签名，不上链

离线签名

payer 用私钥对类型化数据签 EIP-712，得到 (v, r, s)。

谁调谁付 gas

facilitator 拿签名上链 → USDC 自动扣 from → 转 to。Agent 不需要 ETH。

nonce + 时间窗

每个签名只能用一次，过期自动作废。

function transferWithAuthorization(
  address from,
  address to,
  uint256 value,
  uint256 validAfter,
  uint256 validBefore,
  bytes32 nonce,
  uint8 v, bytes32 r, bytes32 s
) external

付款像发 email：签一张授权单，由 seller 兑付。

15 / 33 · LAYERED CHANGE Three layers of change

三层对象，对应三种变化频率

分层让你能只改快的层，不动慢的层。

对象 变化频率 谁来管 回答的问题
PaymentManager 低 季度 / 年 财务与运维 这笔钱是哪条业务线花的？谁来负责？
PaymentInstrument 中 月 / 季度 财务与钱包管理员 这笔钱从哪个钱包、哪条链出？
PaymentSession 高 小时 / 次 业务系统或编排逻辑 这一次能花多少？能花多久？

典型组合：1 个 Manager（业务线）→ 2-3 个 Instrument（不同链/币种）→ 多个 Session（"单次 0.01 USDC" / "本次任务 1 USDC" / "当月 100 USDC"）

16 / 33 · SEPARATION OF DUTIES 职责分离

分层带来的权限分离

对应传统金融合规里的 Separation of Duties 要求。

ManagementRole
"搭架构的人"

CAN创建 Manager / Instrument / Session
CANNOT调 ProcessPayment（显式 DENY）

ProcessPaymentRole
"花钱的人" — 代理拿到的就是这个

CAN在 Session 预算内调 ProcessPayment
CANNOT创建 Session、改预算、换钱包
即使代理被 prompt injection 攻破，能造成的最大损失，也只是当前 Session 的 maxSpendAmount。

对象模型让"用哪条链"靠配置切换 · 协议抽象让"用哪个支付协议"按需扩展。
