// PART 01

"如果硬上"
会发生什么

如果一个团队直接让 Agent 裸用 x402，通常会在四个地方停下来自己造轮子。

四件你必须自己造的轮子

x402 规定了"怎么谈支付"，但没有规定组织内部"怎么安全执行"。后者是 AgentCore Payments 的事。

钱包密钥放哪里
容器、Lambda、Pod 里不能裸放私钥。
KMS / HSM / 自建托管都得走完整生命周期、灾备、合规评审。
支出怎么硬约束
"每次最多 1 USDC"写在 prompt 里，prompt injection 几行就绕过。
真正的预算边界必须由服务端强制。
身份链怎么接
Agent 不是人，传统以用户为中心的 IAM 套不上去。
需要可审计、可撤销、最小权限的代理身份机制。
审计证据怎么留
日志分散在容器、网关、链上不同位置，
事后拼凑很难得到完整链路。

这四件事每一件单独做都是一个工程项目，做齐需要花几个月、搭一支小团队。

新方案的两块拼图

x402 协议 + Amazon Bedrock AgentCore Payments

Piece 1 · Protocol

x402: 复活一个 26 年前的 HTTP 状态码

HTTP 1999 年保留了 402 Payment Required，几乎没人用过。Coinbase 2024 发布 x402，把支付塞进 HTTP 本身——一次交互内完成。

# Step 1+2: client tries → server demands payment
GET /api/article
HTTP/1.1 402 Payment Required
{ accepts: [{ amount: "1000", payTo: "0xF72b...", asset: USDC }] }

# Step 3+4: signs EIP-3009 → retries → server settles → 200
GET /api/article
X-PAYMENT: <base64(signature)>
HTTP/1.1 200 OK + content
X-PAYMENT-RESPONSE: <tx hash>

无账号 · 无 API key · 每次请求当场结清 · USDC on Base ≈ 200 ms · gas ≈ $0.0001

Building Block

关键拼图：EIP-3009 — payer 只签名，不上链

核心机制

离线签名
payer 用私钥对类型化数据签 EIP-712，得到 (v, r, s)
谁调谁付 gas
facilitator 拿签名上链 → USDC 自动扣 from → 转 to
nonce + 时间窗
每个签名只能用一次，过期自动作废

合约函数签名

function transferWithAuthorization(
  address from,        // payer
  address to,          // seller
  uint256 value,
  uint256 validAfter,
  uint256 validBefore,
  bytes32 nonce,
  uint8 v, bytes32 r, bytes32 s
) external

付款像发 email 一样：签一张授权单，由 seller 兑付。Agent 不用管 gas、不用等 mining。

核心对象模型

5 个对象 · 分两期使用 · 三层变化频率

核心对象模型 — 5 个对象

分两期使用：① ② 一次性配置（安装期）· ③ ④ ⑤ 每次支付都用到（运行期）

1
PaymentCredentialProvider
存放钱包凭证 → AWS Secrets Manager · 通过 ARN 引用
install
2
PaymentConnector
钱包后端适配器 · CoinbaseCDP / StripePrivy · 创建 Manager 时焊死
install
3
PaymentManager
账务根节点 · 授权器配置 + 运行时 IAM 角色
runtime
4
PaymentInstrument
钱包引用（非钱包本身）· 绑定一条链 · 私钥永远在 CDP/Privy
runtime
5
PaymentSession
支付上下文 · maxSpendAmount + 有效期 · 超额自动拒绝
runtime

代理运行时只持有 3 个 ID（Manager ARN / Instrument ID / Session ID）— 看不到私钥

三层对象，对应三种变化频率

这就是为什么要分层：只改快的层，不动慢的层。

对象变化频率谁来管回答的问题
PaymentManager 低 季度/年 财务与运维 这笔钱是哪条业务线花的？谁来负责？
PaymentInstrument 中 月/季度 财务与钱包管理员 这笔钱从哪个钱包、哪条链出？
PaymentSession 高 小时/次 业务系统或编排逻辑 这一次能花多少？能花多久？

典型组合：1 个 Manager对应一条业务线 → 下挂 2-3 个 Instrument（不同链/币种） → 每个 Instrument 下按支付场景开不同 Session（"单次 0.01 USDC" / "本次任务上限 1 USDC" / "当月累计 100 USDC"）。

分层带来的权限分离

对应传统金融合规里的职责分离（Separation of Duties）要求。

ManagementRole
CAN: 创建 Manager / Instrument / Session
CANNOT: 调 ProcessPayment（显式 DENY）

"搭架构的人"

ProcessPaymentRole
CAN: 在 Session 预算内调 ProcessPayment
CANNOT: 创建 Session、改预算、换钱包

"花钱的人" — 代理拿到的就是这个

关键设计：代理拿到的是最小权限集。
即使被 prompt injection 攻破，能造成的最大损失也只是当前 Session 的 maxSpendAmount。

对客户意味着什么

从"代理能分析、能建议" 推进到 "代理能自主完成任务"

用 vs 不用 AgentCore Payments

不用 — 自己造

  • 密钥托管：自建 KMS / HSM 流程
  • 预算控制：写 service mesh interceptor
  • 身份绑定：自定义 IAM 桥接
  • 审计日志：拼日志、对链上 tx
  • 3-6 个月 + 一支团队
  • 每加一条新链都要重写
→

用 — 接 API

  • 密钥托管：服务端自动 + Secrets Manager
  • 预算控制：CreatePaymentSession 一行
  • 身份绑定：原生 IAM 集成
  • 审计日志：自动沉淀完整链路
  • 1-2 周接入
  • 多链 / 多钱包通过 Connector 抽象

对客户而言：把"代理能投产付费"的工作量，从数月压缩到数天。

回到最初的四项诉求 — 全部命中

诉求这套方案怎么做
金额可以小，一次交互完成 x402 把支付塞进 HTTP · L2 USDC 单笔成本 ~$0.0001 · 拿条款 / 签名 / 重试 / 收内容一次完成
钱包可管，支出可控 钱包由服务托管 · 私钥永远在 CDP/Privy · maxSpendAmount 服务端硬强制 · 代理改不了上限
身份与影响面可控 ManagementRole vs ProcessPaymentRole 职责分离 · 代理被攻破最大损失 = 当前 Session 上限
决策可追溯 支付决策 / 签名 / 链上 tx hash 自动入 Payments Audit Log + 链上证据 · 不用拼日志

商业意义：让你的 Agent 产品能真正完成"自主调用付费 API、自主购买内容、自主对接其他 Agent"—— 而不是停在"卡在付款墙前等用户点 OK"。

典型客户场景

付费 API 消费
Agent 自主调用市场数据、地图、翻译、专家 LLM等付费 API，按调用计费，每次几美分。
付费内容购买
Agent 替用户买报告、买数据集、买视频片段。卖方挂目录、收 USDC，代理按需购买。
Agent 之间结算
多 Agent 协作时，工作流上游的 Agent 调用下游 Agent 服务，按次实时付款。
SaaS Pay-per-use
把现有 SaaS 接口加一道 x402，让 Agent 用户无需注册就能消费。
RAG 数据采购
Agent 在执行 RAG 时按段落付钱给内容提供者，解决传统订阅与按段计费冲突。
受控自主采购
企业内 Agent 在预算 SLA内自主采购云服务、第三方工具，不需要每次走审批流。

当前 demo 钉死 Base Sepolia 的三处硬编码可改为多链配置。真正约束：facilitator 可用性 + EIP-3009 支持度。

协议 — x402 是第一个，不是唯一

  • MCP 目录里加 payment.protocol 字段
  • 代理侧按字段分发到对应 handler
  • 每种协议（x402 / Stripe / SEPA / L402）单独一个 handler 模块

卖方按内容类型选协议（高频 API 走 x402、订阅走 Stripe、跨境走 SEPA），
代理业务代码不需要重写。

对象模型让"用哪条链"靠配置切换，协议抽象让"用哪个支付协议"按需扩展—— 代理业务代码不用反复重写。
